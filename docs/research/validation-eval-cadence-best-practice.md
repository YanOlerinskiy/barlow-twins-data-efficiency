# Validation / Eval Cadence — Best Practice Survey

*Research-agent report, 2026-06-09. Web-verified against primary sources (paper appendices + official training code/configs + framework docs). Verdicts: **[established]** = stated/measured/coded in a primary source; **[folklore]** = widespread practice, no published proof; **[inference]** = my reasoning from established facts; **[open]** = no evidence found. Companion to `ssl-pretraining-validation-monitoring.md` (what signal to probe), `early-stopping-on-cosine-schedules.md` (when stopping is compatible with the LR schedule), `ssl-training-budgets-steps-vs-epochs.md` and `epoch-budgets-and-lr-scheduling.md` (budget unit / WSD). This report answers a narrower question: **how often to run the val probe, and in what unit (steps vs epochs vs tokens)**, and what that implies for a thesis whose splits span 130× in steps/epoch. Does not re-derive WSD mechanics or the probe protocol (k, τ, db) — see those.*

## Q1. Unit of cadence: steps/iterations vs epochs vs tokens

The field splits cleanly along **single-fixed-dataset vs varying-size/streaming** lines, and the cadence unit tracks the *budget* unit (see `ssl-training-budgets-steps-vs-epochs.md`).

- **Large-scale LM pre-training evaluates in STEPS/TOKENS, never epochs** [established]. **nanoGPT** (`karpathy/nanoGPT/train.py`): `eval_interval = 2000`, `eval_iters = 200`, `log_interval = 1`; the loop is `if iter_num % eval_interval == 0`, i.e. iteration-denominated, and there is no epoch concept (it indexes a token stream). **OLMo** (arXiv:2402.00838, §"Evaluation"): online eval runs *"in-loop every 1000 training steps (or ∼4B training tokens)"*; checkpoints saved ≥ every 1000 steps with naming `step1000-tokens4B`. **Pythia** (arXiv:2304.01373): 154 public checkpoints per model, **step-denominated** (log-spaced early — steps 0,1,2,4,…,512 — then every 1000 steps), explicitly so checkpoints are *comparable across model sizes by step/token*, not epoch. The reason is structural: these corpora are seen ≈once (≤1 epoch), so "epoch" is meaningless and *tokens/steps* is the only stable axis [established].
- **Supervised vision validates PER-EPOCH** [established/folklore]. timm, torchvision ResNet recipes, DeiT/ViT (`facebookresearch/deit`) run a `validate()` call at the end of every epoch; the canonical ImageNet recipe (Goyal 2017, arXiv:1706.02677) is epoch-structured (90 epochs, step-decay at ep 30/60/80). Per-epoch eval is the default because the dataset is **one fixed size**, an epoch is a meaningful unit of "data seen," and one ImageNet epoch is already ~5k–10k steps so per-epoch is fine resolution [inference, well-supported].
- **SSL pre-training mostly does NOT validate online at all** [established] — it checkpoints periodically and evaluates *post hoc*. **DINO** (`facebookresearch/dino/main_dino.py`): `--saveckp_freq default=20`, `--epochs default=100`; saving is `if epoch % saveckp_freq == 0` (epoch-denominated), and **no online kNN** — kNN is a separate `eval_knn.py` run after the fact. **MAE** (`facebookresearch/mae/main_pretrain.py`): loop over epochs, save `if epoch % 20 == 0 or last`, **no validation loop in pre-training**. Barlow Twins / VICReg / MoCo-v3 / SimCLR likewise use a fixed epoch budget and post-hoc probing (confirmed in `ssl-pretraining-validation-monitoring.md` Q1). **online kNN-during-training is a community add-on (lightly `BenchmarkModule`, knn-monitor repos), not in the original FAIR code** [folklore].
- **Principle dictating the choice** [inference, strongly supported by the above]: cadence is denominated in whatever the **budget** is denominated in, and the budget is in *epochs* iff the dataset is a single fixed size seen many times, in *steps/tokens* iff size varies or data streams. A learning-curve study that sweeps dataset size at fixed batch is exactly the "varying size" case — steps/epoch varies, so **epochs are not comparable across splits and steps are the natural unit**. This is the same argument El-Nouby et al. make for *equalizing iterations not epochs* across scales (`ssl-training-budgets-steps-vs-epochs.md`).

## Q2. How OFTEN — the cost / resolution / noise tradeoff

There is **no single published "validate every X" rule**; the binding constraints are framework-encoded overhead bounds and statistical noise.

- **(a) Overhead bound — frameworks make cadence a knob precisely to cap eval cost** [established]. **HF `TrainingArguments`**: `eval_strategy ∈ {no, steps, epoch}`, `eval_steps` (int = #update steps between evals, or float = fraction of total steps) — the *existence* of a steps mode is so you can decouple eval from epoch length on large data. **PyTorch Lightning** `Trainer(val_check_interval=...)`: float (fraction of epoch), **int (every N batches)**, or wall-clock `"DD:HH:MM:SS"`; default `1.0` = once per epoch. **Keras** `ModelCheckpoint(save_freq=...)`: `"epoch"` or an integer batch count. The shared design across all three — an integer *batch/step* mode plus a *fraction-of-epoch* mode — is the field's de facto admission that "per epoch" is too coarse when epochs are huge and you must bound eval cost as a fraction of train time [established that the knob exists; the specific target fraction is folklore].
- **(b) Too-frequent eval on a small val set → noisy selection / val-set overfitting** [established at the principle level]. Each evaluation is a query against the held-out set; selecting the argmax over **many** noisy checkpoint evaluations inflates the chosen score — the **winner's curse / selection bias** (Cawley & Talbot 2010, JMLR 11:2079–2107, *"On Over-fitting in Model Selection…"*: variance of the selection criterion, not just bias, drives over-fitting *in selection*, and the degradation *"can be surprisingly large"*). The more checkpoints you select among, the larger the optimistic bias, and a *small* val set has high per-eval variance → a jagged curve whose argmax is partly noise. With our ~10k-image/200-class TI val probe (`ssl-pretraining-validation-monitoring.md` Q4) this is a real, not hypothetical, concern [established principle; magnitude at our scale **open**].
- **(c) Too-infrequent → miss the peak / poor selection resolution** [inference, well-supported]. Coarse cadence undersamples the val curve, so the recorded argmax can sit far from the true peak and early-stopping fires late; this is the direct motivation for the fraction-of-epoch modes in (a). No paper quantifies the resolution loss for SSL [open].
- **Principled rule that is actually defensible**: choose cadence to (i) *resolve the region of interest* — enough points across the plateau/peak to locate it within the noise — while (ii) keeping eval overhead a small fraction of train time (single-digit %, the implicit target behind framework step-modes) and (iii) not so fine that per-eval noise dominates the between-eval signal. This is a tradeoff statement, **not** a published constant [folklore that "a few % overhead" is the target; the three-way tradeoff is [inference] from (a)–(c)].

## Q3. Checkpoint-selection / early-stopping cadence specifically

- Cadence **does** interact with selection quality, in both directions [established principle, Cawley & Talbot 2010]. *Coarse* cadence → the best-val checkpoint is the best *among the few you happened to sample*, biased away from the true optimum (resolution loss, Q2c). *Fine* cadence on a *noisy* val set → more draws of a noisy max → larger optimistic/winner's-curse inflation of the selected score (Q2b). The selected score is therefore an upward-biased estimate of that checkpoint's true quality, and the bias **grows with the number of checkpoints compared** — so "evaluate as often as possible" is *not* free.
- **Patience should be defined in evaluations, not epochs** [folklore, framework-consistent]: early-stopping callbacks (Keras `EarlyStopping`, Lightning) count *patience* in units of *eval events*, so the meaning of "patience=k" is tied to your cadence — halving cadence halves wall-clock patience. This must be stated explicitly when reporting an early-stop rule.
- **Smoothing the val signal before selection is the standard remedy** for fine-cadence noise [folklore, widely practiced]: select on a moving-average / best-of-smoothed curve rather than the raw argmax; SimCLR-style practice even prefers a *less-noisy metric* (top-5 over top-1) for "best checkpoint" selection precisely because it is more stable (google-research/simclr discussion). No SSL paper formalizes a smoothing-before-selection rule [open].

## Q4. Constant-LR / WSD specific

- Under a **constant (post-warmup) LR stable phase**, every checkpoint sits at the *same* high-ish LR, so the model genuinely *wobbles more step-to-step* than under a decayed LR settling into a minimum [inference from LR dynamics; consistent with `early-stopping-on-cosine-schedules.md` Q5 that constant-LR checkpoints are "similarly-conditioned but unsettled"]. This argues for **smoothing the val curve and/or weight-averaging before selection**, *not* for a different raw cadence.
- The established remedy is **weight averaging along the high/constant-LR trajectory**: **SWA** (Izmailov et al., arXiv:1803.05407) averages weights over a constant/cyclical-LR tail for flatter optima/better generalization; **EMA / LAWA / checkpoint-averaging** (e.g. arXiv:2306.03241 "Early Weight Averaging meets High LR") are used in LLM pre-training specifically because *"high LR … noisy optimizer … final weights may sit in a shallow unstable valley,"* and averaging *"smooths out the outliers."* WSD's cooldown plays a related role: it anneals *after* you pick where to stop, so selection on the stable phase need not coincide with a settled model (`epoch-budgets-and-lr-scheduling.md`, `early-stopping-on-cosine-schedules.md` Q2). **None of these are validated for 1k-image ViT-Tiny Barlow Twins** [open]; they are LLM/large-vision results.
- Net: constant-LR/WSD pushes you toward **(a) smoothing/averaging before you read off "best," and (b) treating the selection point as a region, not a single noisy argmax** — it does not by itself dictate a finer or coarser cadence [inference].

## Q5. Synthesis for this thesis

- **Cadence in STEPS (clamped to a [floor, cap]) is the principled choice over per-epoch** [inference, strongly supported]. The splits span 3 steps/epoch (1k) to ~390 (100k) — 130×. Per-epoch eval would be *3 evals over a whole run* at 1k (uselessly coarse, Q2c) and absurdly frequent relative to signal at 100k. Denominating in steps matches what every varying-size/streaming setup does (nanoGPT/OLMo/Pythia, Q1) and what the budget itself should be denominated in (El-Nouby; `ssl-training-budgets-steps-vs-epochs.md`). A **clamp [floor, cap]** is the right shape because it bounds both failure modes: the *cap* (e.g. ≤500 steps between evals) guarantees enough points to resolve the curve at large N; the *floor* (e.g. ≥50 steps) prevents per-eval noise and probe overhead from dominating at small N where steps/epoch is tiny. The student's existing scheme (≈every 8 epochs, clamped to [50,500] steps) is exactly this pattern and is **defensible**; it is *engineering [folklore]*, not a cited constant.
- **Defensible target for the cap/overhead**: keep total probe cost to a few % of train wall-clock (the implicit framework target, Q2a) *and* ensure ≳20–30 evals across the run so the plateau is resolvable within probe noise [folklore — no published constant; state it as a design choice with the overhead measured, not asserted]. Given the probe is one forward pass over ≤20k images, a few-% overhead easily permits the [50,500] band.
- **Smoothing the val curve before checkpoint selection is the standard remedy for fine-cadence + noisy-probe** and should be adopted [folklore, framework-consistent]: select on a moving-average/best-of-smoothed kNN curve (and/or report the selected score with the winner's-curse caveat, Q2b/Q3). Under the constant-LR/WSD schedule this is doubly motivated (Q4); optionally corroborate with a label-free signal (RankMe, see `ssl-pretraining-validation-monitoring.md` Q2).
- **Rigor caveats to keep the thesis honest**: (i) the steps-vs-epochs *convention* is [established] but its load-bearing evidence is LLM/large-vision — do **not** claim a measured benefit at 1k-image SSL [open]; (ii) the [50,500] numbers and the "few-% overhead / ≥20 evals" targets are **[folklore]** design choices, not cited results — justify them by *measured* overhead and *observed* curve resolution on one split, not by citation; (iii) winner's-curse inflation of the selected kNN score is a [established] *mechanism* but its magnitude on a 10k/200-class probe is [open] — report it as a threat to validity, and prefer a *smoothed* selection to blunt it; (iv) patience, if used, must be reported in **evaluations** with the cadence stated (Q3).
