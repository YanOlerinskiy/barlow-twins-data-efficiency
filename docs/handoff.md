Three things to keep in mind for v2:

1. **Big recovery at small splits** (~+7-8 pp at 1k/2k) — confirms the over-training hypothesis quantitatively.
2. **Diminishing recovery as data grows** — exactly the pattern predicted.
3. **At 16k the early-stop run is slightly *worse* than equal-compute.** Almost certainly the patience-3 trigger firing too early at scales where the model is still legitimately improving. Worth investigating: tighten `min_delta`, raise patience, or scale them with split size before publishing the recovery numbers.

## Recent design decisions, with rationale (so they don't get re-litigated)

- **lightly over solo-learn / Facebook ref**: lightly's loss + projector + transforms are a clean drop-in for our HF ViT backbone. solo-learn is Hydra+Lightning (too invasive). Facebook ref is hardcoded for ResNet-50 + LARS + 8-GPU DDP.
- **λ = 1e-2** (paper: 5e-3 with D=8192; ours: 1e-2 with D=1024). Rule: keep λ·D in a comparable order of magnitude (paper ≈ 41, ours ≈ 10). Mentioned in CLAUDE.md.
- **Projector 192 → 1024 → 1024**, lightly's default 3-layer structure. Paper's 8192 would be larger than the encoder itself.
- **min_scale=0.25** for `RandomResizedCrop` (paper / lightly default 0.08). At 64×64, 0.08 gives ~18×18 crops which are degenerate and trigger collapse.
- **GaussianBlur disabled** on both views. The paper's 23-pixel kernel covers >⅓ of a 64×64 image.
- **Asymmetric solarization preserved** (view 1 p=0, view 2 p=0.2) — the BT/BYOL signature.
- **AdamW lr=3e-4 over LARS**: LARS is for batch 2048+; we run at batch 256. AdamW is the modern ViT-SSL standard.
- **Linear probe = SGD + cosine + 100 ep + best-of-100, with horizontal flip on train**: Caron et al. 2021 App. B.2 / MoCo v3 `main_lincls.py` recipe. Mild deviation from B.2 (the flip on train), team-agreed. Documented in `docs/LINEAR_PROBE_PROTOCOL.md`.
- **Linear probe operates on `final` checkpoint only**, not `max(final, best_knn)`. The earlier sklearn-LR + best-of-checkpoints protocol inflated small splits by ~5-10 pp via selection leakage. Don't go back to it for headline numbers.
- **kNN diagnostic during equal-compute pretraining used CIFAR-10 test as the query set.** That's mild leakage (the test set was *touched* during pretraining), but the signal was never used for stopping or selection in the equal-compute run. Disclosed honestly in the paper.
- **Early-stop kNN uses a held-out CIFAR-10 train val** (`get_knn_db_val_loaders(db_size=5000, val_size=5000, stratified=True)`): 500 per class for db, next 500 per class for val. CIFAR-10 test untouched during pretraining.
- **Single seed (42 pretrain, 0 eval) throughout.** No multi-seed runs. Run-to-run GPU non-determinism is ~1-2 pp at split=32k for identical seeds. Multi-seed planned for v2.

## What's open for v2

In priority order:

1. **Group-aggregated learning curve** (Week 6 milestone, may have slipped). All 5 methods on one plot, drawn from `evaluation/results.csv`. Yan's numbers are ready and in the shared CSV format; depends on teammates submitting theirs.
2. **Convergence-based stopping** — implementation is done, but the 16k regression must be diagnosed before publishing. Suspected causes: patience-3 too aggressive when knn_every is small relative to the schedule, min_delta too coarse for late-training tiny gains. Re-run with `patience=5` or `min_delta=0.003` on 16k as a controlled test.
3. **Multi-seed runs**: 3 seeds × 8 splits = 24 runs per method. Should be ~2-3× current per-run wall-clock, so ~6-8 hours of compute, ~$1-2 on vast.ai.
4. **Augmentation extensions**: multi-crop (DINO-style; ~+2-5 pp expected), RandAugment, view-level Mixup. The Barlow Twins paper's discussion section explicitly invites augmentation extensions. Multi-crop has the biggest expected lift but requires position-embedding interpolation for the smaller local crops at 64×64.
5. **CIFAR-100 second downstream task**: drop-in via `torchvision.datasets.CIFAR100`; ~10 lines of code in `leo_protocol_eval.py` plus a `--dataset` flag; ~90 min of compute. Tests whether the data-efficiency claim generalises across class counts.
6. **PEFT fine-tuning** instead of linear probe — planned per CLAUDE.md. Typically +5-15 pp over linear probe. Different lens on data efficiency.
7. **Confusion-matrix and per-class accuracy analysis**: where does the model fail? See the corresponding meeting notes / poster discussions. ~30 lines of additional eval code; no new compute needed (re-uses existing checkpoints).

## Useful environment / gotcha list

- **SSH agent forwarding inside tmux**: VS Code's `SSH_AUTH_SOCK` becomes stale after reconnect. Fix is the bashrc symlink trick (already documented; apply to any new vast.ai box):
  ```bash
  if [ -n "$SSH_AUTH_SOCK" ] && [ -S "$SSH_AUTH_SOCK" ] && [ ! -L "$SSH_AUTH_SOCK" ]; then
      ln -sf "$SSH_AUTH_SOCK" "$HOME/.ssh/agent-sock"
  fi
  export SSH_AUTH_SOCK="$HOME/.ssh/agent-sock"
  ```
- **HF dataset cache**: `export HF_HOME=/workspace/hf_cache` on rented boxes — the dataset persists across container rebuilds.
- **vast.ai SSH key must be added BEFORE renting**, not after (otherwise requires instance stop/start).
- **Always rsync results + checkpoints before destroying** the vast.ai instance. The disk is wiped on destroy.
- **W&B project**: `yan-olerinskiy-tu-delft/barlow-twins-data-efficiency`. Free tier, browser monitoring works fine from phone.
- **PyTorch in `requirements.txt`** (the shared one, not `requirements_barlow_twins.txt`) has fake version pins (`torch==2.11.0` doesn't exist). Don't touch that file — it's group-owned — but on every fresh box install from `requirements_barlow_twins.txt` instead.
